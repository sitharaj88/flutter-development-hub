# Capstone Execution Guide

The capstone should feel like a small real product, not a random final assignment. This guide helps turn the final project into a more meaningful learning and proof artifact.

## What the capstone should achieve

- combine multiple training stages into one coherent app
- test whether the learner can plan before building
- show whether the learner can explain decisions clearly
- produce something presentable and improvable

## Good capstone workflow

1. choose a scope that can be completed well
2. map the user flow before building
3. define reusable UI pieces and data requirements
4. build in checkpoints rather than waiting until the end
5. review both design and engineering quality

## Strong capstone ingredients

- at least one meaningful user flow
- complete UI states
- simple but explainable architecture
- cleaner project structure
- a clear explanation of what was built and why

## Presentation guidance

- explain the app problem clearly
- show the main user flow
- discuss design choices and technical tradeoffs
- describe what would be improved next with more time

## What reviewers should look for

- clarity of scope
- completeness of flow
- quality of structure
- confidence of explanation
- signs of real understanding rather than copied implementation
